## Is bringing your own tool right for your work?

You may benefit from bringing a tool to BDC if you:

- Have a script you want to run or reuse across multiple files
- Have a process or analysis that is expected to take several hours
- Need to formalize and scale a multi-step workflow
- Have dependencies that would benefit from containerization
- Want to reuse or integrate an existing Dockerized tool
- Want application versioning

## Before You Start

Bringing tools into BDC typically requires:

* Command-line tools or scripts familiarity (R, Python, Bash)  
* Understanding of inputs, outputs, and dependencies for each script/tool/workflow  
* Ability to identify the software, packages, and runtime environment your tool needs to run

### **Compatibility Considerations**

Not all tools are immediately compatible. Before getting started, consider:

* **Data compliance and security**: Tools must meet requirements for controlled-access data and secure environments.  
* **Licensing**: Tools that depend on proprietary software (for example, MatLab) or have licensing restrictions are not supported.  
* **Interactive tools**: Tools or workflows that rely on interactive inputs or provide interactive visualizations are not typically supported through standard, self-service workflows. These use cases may be supported through dedicated, funded projects within BDC.  
* **Containerization**: Tools must be containerizable and describable in supported workflow languages. Thus tools requiring direct hardware access, always-on services, and tools with no defined input are not supported.  
* **Nested containers**: Tools that require one Docker container to call or run another Docker container are not supported. Workflows may include multiple tools, with each tool using its own container
* **Dependencies**: Tools with unavailable or non-portable dependencies are not supported.
